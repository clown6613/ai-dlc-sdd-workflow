# SDD ワークフロー 手順書

**対象:** Claude Code を使ったチーム開発  
**前提:** [AI-DLC 基礎編](../docs/01-fundamentals.md)・[使い分けガイド](../docs/03-approaches.md)を読んでいること

---

## 全体フロー

```
Phase 1: Intent 作成          ← 人間が主導
Phase 2: Spec 生成・承認       ← AI が生成 → 人間が承認（Mob Elaboration）
Phase 3: タスク実装            ← AI が実装 → 人間が確認
Phase 4: 品質ゲート            ← 自動チェック
Phase 5: レビュー・マージ       ← 人間が承認（Mob Construction）
```

---

## Phase 1: Intent 作成

**誰が:** PdM または担当エンジニア  
**所要時間:** 15〜30分

### 手順

1. `specs/<feature-name>/` ディレクトリを作成する
2. `templates/intent.md` をコピーして `specs/<feature-name>/intent.md` として保存する
3. 以下の3要素をすべて記述する:

   **Description（必須）:**  
   「何を」「誰のために」「どんな目的で」を1〜2文で。  
   例: `レポート画面に CSV ダウンロード機能を追加する。マーケターが月次データを手元で分析できるようにするため。`

   **Context（必須）:**  
   - ビジネス背景（なぜ今必要か）
   - 技術的制約（既存システム、パフォーマンス要件）
   - スコープ外（今回はやらないこと）

   **Completion Criteria（必須）:**  
   - 成功ケース（正常系）
   - 失敗ケース（エラー系・境界値）
   - 品質ゲート（テスト・lint・型チェック）

4. Intent をチームでレビューする（5分程度。スコープのずれを早期発見）

### チェックリスト
- [ ] Description が1〜2文で書かれているか
- [ ] Context にスコープ外が明記されているか
- [ ] Completion Criteria に失敗ケースが含まれているか

## Phase 2: Spec 生成・承認（Mob Elaboration）

**誰が:** AI が生成 → チームが承認  
**所要時間:** Spec 生成 5〜10分 + Mob Elaboration 15〜30分

### 手順

1. Claude Code で以下を実行:
   ```
   /sdd-spec specs/<feature-name>/intent.md
   ```

2. AI が `specs/<feature-name>/spec.md` を生成する（3ステージ）:
   - Stage 1: 要件（EARS記法）
   - Stage 2: 設計
   - Stage 3: タスクリスト（SMAV）

3. **Mob Elaboration を実施する:**  
   チームで spec.md をレビューし、以下を確認:
   - 要件に漏れ・誤りがないか
   - 設計がアーキテクチャと整合しているか
   - タスクが適切な粒度に分割されているか
   - 完了基準が測定可能か

4. 修正が必要な場合: spec.md を直接編集するか、Claude Code に修正を依頼する

5. チームが承認したら Spec の `ステータス` を `Approved` に変更する

> **重要:** Spec が承認されるまで実装を開始しない。これが SDD の核心です。

### スクラムチームの場合

| スクラムセレモニー | SDD での役割 |
|-----------------|------------|
| スプリントプランニング | Intent 確認 + Spec の Mob Elaboration |
| バックログリファインメント | Intent の品質改善（Completion Criteria の精緻化） |

---

## Phase 3: タスク実装

**誰が:** AI が実装 → 人間が確認  
**所要時間:** タスクによる（数時間〜1日）

### 手順

1. Claude Code で以下を実行（実装開始を宣言）:
   ```
   specs/<feature-name>/spec.md の Task 1 から実装を開始します
   ```

2. AI は Spec のタスクリストを**上から順に1タスクずつ**実装する

3. 各タスク完了後:
   - AI が品質チェックコマンドを実行する
   - テスト・型チェック・lint がすべて通過したら次のタスクへ
   - 問題があれば修正してから次へ進む

4. AI が仕様から逸脱する可能性がある場合は**実装前に確認を求める**

5. 疑問点が生じたら: AI がコアメンタルモデルの「質問」ステップで確認する
   ```
   例: "Spec では 404 を返すよう指定されていますが、
        既存コードでは 400 を返している箇所があります。
        どちらに統一しますか？"
   ```

### 実装中の注意点

- Spec にないことは**勝手に実装しない**（スコープクリープ防止）
- 実装途中で仕様の誤りを発見した場合: spec.md を修正してから実装を続ける
- タスクが大きすぎると感じたら: 分割を提案する

---

## Phase 4: 品質ゲート

**誰が:** 自動（hooks）または手動  
**所要時間:** 数分

### 自動実行（hooks 設定済みの場合）

ファイル保存時に自動で以下が実行されます（`.claude/settings.json` で設定）:
- TypeScript ファイル変更時: 型チェック
- ソースファイル変更時: lint

### 手動実行

実装完了後、以下をすべて実行して通過を確認する:

```bash
# テスト
npm test

# 型チェック（TypeScript の場合）
npx tsc --noEmit

# lint
npx eslint . --max-warnings 0

# フォーマット確認
npx prettier --check .
```

### 品質ゲートが失敗した場合

1. エラーメッセージを確認する
2. Claude Code に修正を依頼する（`/sdd-review` でコンテキストを提供できる）
3. 修正後、再度品質ゲートを実行する

---

## Phase 5: レビュー・マージ（Mob Construction）

**誰が:** 人間がレビュー・承認  
**所要時間:** 15〜30分

### 手順

1. Claude Code で仕様適合チェックを実行:
   ```
   /sdd-review specs/<feature-name>/spec.md
   ```

2. AI が spec.md と実装を比較し、以下をレポートする:
   - 各タスクの完了状況（✅/❌）
   - 仕様との不整合箇所
   - テストカバレッジ状況

3. **Mob Construction を実施する:**  
   チームで実装レビューを行い、以下を確認:
   - Spec のすべてのタスクが完了しているか
   - 実装が仕様の意図と合っているか
   - 品質ゲートがすべて通過しているか
   - 意図していない変更が含まれていないか

4. 問題があれば Phase 3 に戻る

5. 承認後、PR を作成してマージする

> **デプロイは必ず人間が承認する。** AI が自動でデプロイすることは禁止です。

## ファイル構造

```
specs/
└── <feature-name>/
    ├── intent.md     ← Phase 1: 作成（人間）
    └── spec.md       ← Phase 2: AI 生成 → 人間承認

.claude/
├── settings.json     ← hooks 設定
└── commands/
    ├── sdd-spec.md   ← /sdd-spec コマンド
    └── sdd-review.md ← /sdd-review コマンド

templates/
├── intent.md         ← コピー元テンプレート
└── spec.md           ← Spec の構造テンプレート
```

## Human in the Loop の判断基準

| アクション | パターン | 理由 |
|-----------|---------|------|
| Spec の承認 | 全件レビュー | 実装の方向性を決める重要な判断 |
| 通常の機能実装 | スコアベース（テスト通過 + lint クリア） | 品質ゲートで自動検証可能 |
| セキュリティ関連の実装 | 全件レビュー | リスクが高く、自動検証困難 |
| 本番デプロイ | 全件レビュー（必須） | 取り消し困難な操作 |
| 定型的な修正（依存更新等） | スコアベース → 将来的にエスカレーション | 影響範囲が限定的 |

## よくある質問

**Q: Intent を書くのに時間がかかる**  
A: 最初は10〜15分かかりますが、テンプレートを使えば慣れると5〜10分になります。Intent を明確にする時間は、後の手戻りを防ぐ投資です。

**Q: Spec が長すぎて読むのが大変**  
A: タスクの複雑さに応じて深さを調整してください（「柔軟な深さ」の原則）。5行の修正なら SMAV 完了基準のみでも十分です。

**Q: スクラムのスプリントと Bolt の関係は？**  
A: 1スプリントの中に複数の Bolt（Spec + 実装サイクル）が入ります。スプリントプランニングで各 Bolt の Intent を確認し、Mob Elaboration で Spec を承認するのが自然な流れです。

**Q: 一人開発でも SDD は使えるか**  
A: 使えます。Mob Elaboration/Construction を「自分が AI の提案を承認するステップ」として一人でも実施できます。Intent と Spec を書くことで、AI との対話の品質が上がります。
