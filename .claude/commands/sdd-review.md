# /sdd-review — 実装を Spec と照合してレビューする

**使い方:**
- `/sdd-review specs/<feature>/spec.md` — パスを直接指定
- `/sdd-review` — 引数なしで実行すると、最近作成された Spec の一覧から選べる

---

## このコマンドがすること

指定された `spec.md` の内容と現在の実装を比較し、仕様適合状況をレポートします。

1. `spec.md` を読み込み、タスクリストと完了基準を抽出する
2. タスク内容をもとに関連ファイルをコードベースから探す
3. 品質ゲートコマンドを実際に実行して結果を取得する
4. 各タスクの完了状況を判定する
5. Mob Construction のための判断材料をレポートとして出力する

---

## 実行手順

### Step 0: 対象の spec.md を決める

`$ARGUMENTS` が指定されていればそのパスを使う。

**指定がない場合:**  
次のコマンドで `specs/` 以下の `spec.md` を更新日時の新しい順に取得する:

```bash
find specs/ -name "spec.md" | xargs ls -t 2>/dev/null
```

結果を次の形式で提示し、番号で選んでもらう:

> `specs/` 内の Spec（新しい順）:
>
> 1. `specs/user-auth/spec.md`（10分前）
> 2. `specs/push-notification/spec.md`（昨日）
> 3. `specs/profile-edit/spec.md`（3日前）
>
> 番号を選ぶか、別のパスを入力してください。

`specs/` が空または存在しない場合:

> `specs/` に Spec が見つかりません。  
> 先に `/sdd-spec` で Spec を作成してください。

---

### Step 1: spec.md を読み込む

確定したパスの `spec.md` を Read ツールで読み込む。  
Stage 3 のタスクリストから以下を抽出する:
- 各タスクの名前・内容・成功条件
- 品質チェックコマンド（`npm test` / `yarn test` / `tsc` / `eslint` 等）

---

### Step 2: 関連ファイルを探す

タスクの内容・成功条件に登場するキーワード（クラス名・関数名・エンドポイント・画面名など）を使って、コードベースから関連ファイルを探す。

**探し方（この順で試みる）:**

```bash
# 1. タスクに登場するキーワードで grep
grep -r "[キーワード]" src/ --include="*.ts" --include="*.tsx" -l

# 2. 機能名に対応するファイル名で find
find src/ -name "*[機能名に関連する単語]*" -type f

# 3. テストファイルを確認
find src/ -name "*.test.ts" -o -name "*.spec.ts" | xargs grep -l "[キーワード]" 2>/dev/null
```

見つかったファイルを Read ツールで読み込み、各タスクの実装状況を確認する。

---

### Step 3: 品質ゲートを実行する

spec.md の品質チェックコマンドを実際に実行して結果を取得する。

**実行する順序:**

```bash
# テスト（package.json の scripts を確認して適切なコマンドを選ぶ）
npm test 2>&1 || yarn test 2>&1

# 型チェック（tsconfig.json が存在する場合）
npx tsc --noEmit 2>&1

# lint
npx eslint . --max-warnings 0 2>&1
```

コマンドが存在しない・設定がない場合はスキップして「未設定」と記録する。

---

### Step 4: レポートを出力する

以下のフォーマットでレポートを出力する:

```markdown
## SDD レビューレポート: [機能名]

**Spec:** specs/[機能名]/spec.md  
**レビュー日時:** [今日の日付]

---

### タスク完了状況

| タスク | ステータス | 根拠 |
|--------|-----------|------|
| Task 1: [名前] | ✅ 完了 | [該当ファイルと確認した内容] |
| Task 2: [名前] | ⚠️ 部分完了 | [実装はあるが成功条件を一部満たしていない箇所] |
| Task 3: [名前] | ❌ 未実装 | [対応するコードが見つからなかった] |

---

### 品質ゲート実行結果

| チェック | 結果 | 詳細 |
|---------|------|------|
| テスト | ✅ 通過 / ❌ 失敗 / ⚠️ 未設定 | [失敗した場合はエラー内容を要約] |
| 型チェック | ✅ 通過 / ❌ 失敗 / ⚠️ 未設定 | [型エラーがある場合はファイル名と行番号] |
| lint | ✅ 通過 / ❌ 失敗 / ⚠️ 未設定 | [エラーがある場合はルール名と件数] |

---

### 仕様との不整合

[不整合がある場合は具体的に記述。例:]
- Task 2: Spec では 401 を返すべきところ、実装では 400 を返している（AuthController.ts:42）
- スコープクリープ: Spec にない「パスワード強度チェック」が実装されている（LoginScreen.tsx:87）

[不整合がない場合: 「仕様との不整合なし」]

---

### Mob Construction での確認ポイント

[チームが目視で確認すべき箇所。自動チェックで検出できないもの。例:]
- エラーメッセージの文言が UX 観点で適切か
- ローディング中のUI体験
- エッジケース（ネットワーク復帰時の挙動）

---

### 判定

- [ ] すべてのタスクが完了している
- [ ] 品質ゲートがすべて通過している
- [ ] 仕様との重大な不整合がない
- [ ] スコープクリープがない

→ **マージ可能 / 要修正**

[要修正の場合は修正すべき箇所を箇条書きで列挙]
```

---

## 注意

- 関連ファイルが見つからないタスクは「❌ 未実装（ファイル未検出）」と記録し、手動確認を促す
- スコープクリープ（Spec にない実装）は必ず指摘する
- セキュリティ上の懸念（認証バイパス・機密情報のログ出力など）は最優先で明示する
- 品質ゲートコマンドの実行に失敗した場合（コマンドが存在しない等）は「⚠️ 未設定」として記録し、手動実行を促す
