# /sdd-review — 実装を Spec と照合してレビューする

**使い方:** `/sdd-review [spec.md のパス]`  
例: `/sdd-review specs/csv-download/spec.md`

---

## このコマンドがすること

指定された `spec.md` の内容と現在の実装を比較し、仕様適合状況をレポートします。

1. `spec.md` を読み込み、タスクリストと完了基準を抽出する
2. 現在のコードベースを確認する
3. 各タスクの完了状況を判定する
4. 仕様との不整合があれば具体的に指摘する
5. Mob Construction のための判断材料を提供する

## 実行手順

1. `$ARGUMENTS` のパスの `spec.md` を Read ツールで読み込む
2. Stage 3 のタスクリストと完了基準を抽出する
3. 関連するソースファイル・テストファイルを確認する
4. 以下のフォーマットでレポートを出力する

## レポートフォーマット

```markdown
## SDD レビューレポート: [機能名]

### タスク完了状況

| タスク | ステータス | 備考 |
|--------|-----------|------|
| Task 1: [名前] | ✅ 完了 / ❌ 未完了 / ⚠️ 部分完了 | [詳細] |
| Task 2: [名前] | ✅ 完了 / ❌ 未完了 / ⚠️ 部分完了 | [詳細] |

### 品質ゲート状況

| チェック項目 | ステータス |
|------------|-----------|
| テスト通過 | ✅ / ❌ |
| 型チェック | ✅ / ❌ |
| lint | ✅ / ❌ |

### 仕様との不整合

[不整合がある場合はここに具体的に記述]
[不整合がない場合: 「仕様との不整合なし」]

### Mob Construction での確認ポイント

[チームが特に注意して確認すべき箇所]

### 判定

- [ ] すべてのタスクが完了している
- [ ] 品質ゲートが通過している
- [ ] 仕様との重大な不整合がない
→ **マージ可能 / 要修正**
```

## 注意

- レビューの結果が「要修正」の場合、修正箇所を具体的に示す
- 仕様に記載のない実装（スコープクリープ）も指摘する
- セキュリティ上の懸念がある場合は必ず明示する
