実装またはタスク作成の前に、曖昧な部分・未決定の設計・リスクとなるエッジケースを特定し、一問ずつ質問して解消する。

質問は一度に一問だけ聞く。

各質問には推奨答えを提示する。

コードベースを調べれば答えられる質問（既存の実装・型定義・API設計・認証方式など）は自分で調べて答え、ユーザーには聞かない。

## 調査する観点（この順で重要度が高い）

1. **スコープ境界** — 何をやって何をやらないか。「も対応しますか？」という判断
2. **既存コードとの整合性** — 既存の実装・命名・アーキテクチャに沿っているか
3. **エラーハンドリング** — 失敗ケース・境界値・タイムアウトの扱い
4. **ユーザー体験** — エラー時の表示・ローディング状態・画面遷移
5. **技術的制約** — 認証方式・DB設計・パフォーマンス要件
6. **セキュリティ** — 認証・認可・入力バリデーション

## 終了

重要な曖昧点がすべて解消されたら、以下のサマリーを出力して終了する。

```
## 明確になった内容

### 決定事項
- [決定した内容]

### スコープ外（確認済み）
- [やらないと決めた内容]

### 次のステップ
→ この内容を intent.md に反映してから /sdd-spec を実行してください
```
