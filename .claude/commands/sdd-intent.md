# /sdd-intent — 対話形式で Intent を作成する

**使い方:** `/sdd-intent`  
引数は不要です。まず自由に話してください。

---

## このコマンドの動作

自然言語で作りたいものを話してもらい、そこから機能名を決めて Intent を完成させます。  
最初の説明で既に答えが出ている質問はスキップします。

---

## 実行手順

### Step 0: 自由記述を受け取る

最初に次のメッセージを表示する（必ずこの文言で始める）:

> 作りたいもの・直したいこと・得たい成果を自由に教えてください。  
> 箇条書きでも、思いつくままでも構いません。

ユーザーの回答を受け取る。

---

### Step 1: 既存の specs/ を確認してから機能名を提案

**まず `specs/` ディレクトリの中身を確認する。**  
`ls specs/` または `find specs/ -maxdepth 1 -type d` を実行して、既に存在する機能名の一覧を取得する。

その上で Step 0 の内容を分析し、以下を提示する:

> **理解した内容:**  
> [Step 0 の内容を1〜2文でサマリー]
>
> **機能名の候補:** `[kebab-case で提案]`  
> **保存先:** `specs/[機能名]/intent.md`
>
> この機能名でよいですか？変更したい場合は別の名前を教えてください。

**注意:** 提案する機能名が既に `specs/` に存在する場合は、そのまま提案せずに次のように伝える:

> `specs/[候補名]/` は既に存在します。  
> 別の名前を提案します: `[別の候補]`  
> または、既存の Intent を上書きする場合は「上書き」と答えてください。

ユーザーが承認または別名を提示したら、確定した機能名を保存先として使う。

---

### Step 2〜6: 不足情報の収集

Step 0 の内容を分析して、**まだ答えが得られていない項目だけ** を1問ずつ聞く。  
既に十分な情報がある項目はスキップしてよい。

収集する項目と判断基準:

| 項目 | スキップ条件 |
|------|-------------|
| **Description**（何を・誰のために・なぜ） | Step 0 で目的と対象が明確に述べられていた場合 |
| **ビジネス背景**（なぜ今必要か） | Step 0 で背景・課題が述べられていた場合 |
| **技術的制約**（既存システム・スタック・要件） | Step 0 で言及があった場合 |
| **スコープ外**（今回やらないこと） | Step 0 で明言されていた場合、または明らかに不要な場合 |
| **成功ケース**（正常系の動作） | Step 0 で具体的な完成イメージが述べられていた場合 |
| **失敗ケース・品質ゲート** | ほぼ常に聞く（省略されることが多い） |

**各質問の聞き方（例を必ず含める）:**

ビジネス背景が不明な場合:
> **なぜ今この機能が必要ですか？**
>
> ビジネス上の背景・課題・誰が困っているかを教えてください。  
> 例: `マーケターから「Excelで分析したい」という要望が増えている`  
> 例: `毎月手動でデータを抽出している作業を自動化したい`  
> （わからない・特にない場合は「なし」と答えてください）

技術的制約が不明な場合:
> **技術的な制約や、既存システムとの関係はありますか？**
>
> 例: `バックエンドは Node.js + PostgreSQL`  
> 例: `既存の認証は Cognito を使っている`  
> 例: `レスポンスは p95 で 500ms 以内に収めたい`  
> （なければ「なし」で OK です）

スコープ外が不明な場合:
> **今回やらないこと（スコープ外）はありますか？**
>
> 明示しておくと実装時のブレを防げます。  
> 例: `リアルタイム集計は今回やらない`  
> 例: `スマホ対応は次フェーズ`  
> 例: `Excel 形式（.xlsx）は対象外`  
> （なければ「なし」で OK）

成功ケースが不明な場合:
> **どうなれば「完成」ですか？正常系の動作を教えてください。**
>
> ユーザーが期待する動作を具体的に。複数あれば箇条書きで。  
> 例:
> - `CSV ファイルがダウンロードされる`
> - `日次・月次の切り替えができる`
> - `ファイル名に対象期間の日付が入る`

失敗ケース・品質ゲートが不明な場合:
> **最後に、エラーケースと品質基準を教えてください。**
>
> **エラーケース:** どんな失敗を考慮しますか？  
> 例: `存在しない ID → 404`、`タイムアウト（60秒超）→ 504`、`データ 0 件 → ヘッダー行のみの CSV`
>
> **品質ゲート:** どの自動チェックを通過させますか？  
> 例: `単体テスト通過`、`TypeScript 型エラーゼロ`、`ESLint クリア`  
> （特になければ「デフォルト」と答えてください → テスト・型チェック・lint 通過を設定します）

---

### Step 7: intent.md の生成

収集した全情報をもとに `specs/[機能名]/intent.md` を生成する。  
以下の形式で出力する:

```markdown
# Intent: [機能名を日本語で表現]

## Description（何を作るか）
[Descriptionの内容]

## Context（なぜ作るか）

### ビジネス背景
[ビジネス背景の内容。なければ省略]

### 技術的制約
[技術的制約の内容。なければ省略]

### スコープ外
[スコープ外の内容。なければ省略]

## Completion Criteria（完了の定義）

### 成功ケース
- [成功ケース1]
- [成功ケース2]

### 失敗ケース
- [失敗ケース1]
- [失敗ケース2]

### 品質ゲート
- [品質基準1]
- [品質基準2]

---

## メタデータ

| 項目 | 内容 |
|------|------|
| 作成日 | [今日の日付] |
| 関連 Issue/チケット | — |
| 優先度 | — |
| 見積もり（Bolt） | — |
```

生成後に次のメッセージを表示する:

> ✅ `specs/[機能名]/intent.md` を作成しました。
>
> 内容を確認して、必要があれば編集してください。  
> 準備ができたら `/sdd-spec specs/[機能名]/intent.md` で Spec を生成できます。
