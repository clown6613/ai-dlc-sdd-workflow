# /sdd-intent — 対話形式で Intent を作成する

**使い方:** `/sdd-intent`  
引数は不要です。まず自由に話してください。

---

## このコマンドの動作

自然言語で作りたいものを話してもらい、そこから機能名を決めて Intent を完成させます。  
最初の説明で既に答えが出ている質問はスキップします。

---

## 実行手順

### Step 0: 自由記述を受け取る

最初に次のメッセージを表示する（必ずこの文言で始める）:

> 作りたいもの・直したいこと・得たい成果を自由に教えてください。  
> 箇条書きでも、思いつくままでも構いません。

ユーザーの回答を受け取る。

---

### Step 1: 機能名の提案・確認

Step 0 の内容を分析して、以下を提示する:

> **理解した内容:**  
> [Step 0 の内容を1〜2文でサマリー]
>
> **機能名の候補:** `[kebab-case で提案]`  
> **保存先:** `specs/[機能名]/intent.md`
>
> この機能名でよいですか？変更したい場合は別の名前を教えてください。

ユーザーが承認または別名を提示したら、確定した機能名を保存先として使う。

---

### Step 2〜6: 不足情報の収集

Step 0 の内容を分析して、**まだ答えが得られていない項目だけ** を1問ずつ聞く。  
既に十分な情報がある項目はスキップしてよい。

収集する項目と判断基準:

| 項目 | スキップ条件 |
|------|-------------|
| **Description**（何を・誰のために・なぜ） | Step 0 で目的と対象が明確に述べられていた場合 |
| **ビジネス背景**（なぜ今必要か） | Step 0 で背景・課題が述べられていた場合 |
| **技術的制約**（既存システム・スタック・要件） | Step 0 で言及があった場合 |
| **スコープ外**（今回やらないこと） | Step 0 で明言されていた場合、または明らかに不要な場合 |
| **成功ケース**（正常系の動作） | Step 0 で具体的な完成イメージが述べられていた場合 |
| **失敗ケース・品質ゲート** | ほぼ常に聞く（省略されることが多い） |

**聞き方の例:**

成功ケースが不明な場合:
> **どうなれば「完成」ですか？**  
> ユーザーが期待する動作を具体的に教えてください。  
> 例: 「CSV がダウンロードされる」「日次・月次で切り替えられる」

失敗ケース・品質ゲートが不明な場合:
> **最後に、エラーケースと品質基準を教えてください。**
>
> - どんな失敗を考慮しますか？（例: ID が存在しない → 404）
> - 自動チェックの基準は？（例: テスト通過・型エラーゼロ・lint クリア）  
>   特になければ「デフォルト」と答えてください。

---

### Step 7: intent.md の生成

収集した全情報をもとに `specs/[機能名]/intent.md` を生成する。  
以下の形式で出力する:

```markdown
# Intent: [機能名を日本語で表現]

## Description（何を作るか）
[Descriptionの内容]

## Context（なぜ作るか）

### ビジネス背景
[ビジネス背景の内容。なければ省略]

### 技術的制約
[技術的制約の内容。なければ省略]

### スコープ外
[スコープ外の内容。なければ省略]

## Completion Criteria（完了の定義）

### 成功ケース
- [成功ケース1]
- [成功ケース2]

### 失敗ケース
- [失敗ケース1]
- [失敗ケース2]

### 品質ゲート
- [品質基準1]
- [品質基準2]

---

## メタデータ

| 項目 | 内容 |
|------|------|
| 作成日 | [今日の日付] |
| 関連 Issue/チケット | — |
| 優先度 | — |
| 見積もり（Bolt） | — |
```

生成後に次のメッセージを表示する:

> ✅ `specs/[機能名]/intent.md` を作成しました。
>
> 内容を確認して、必要があれば編集してください。  
> 準備ができたら `/sdd-spec specs/[機能名]/intent.md` で Spec を生成できます。
