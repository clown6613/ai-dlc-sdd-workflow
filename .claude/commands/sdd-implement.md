# /sdd-implement — Spec のタスクを順番に実装する

**使い方:**
- `/sdd-implement specs/<feature>/spec.md` — パスを直接指定
- `/sdd-implement` — 引数なしで実行すると、最近承認された Spec の一覧から選べる

---

## このコマンドがすること

承認済みの `spec.md` を読み込み、タスクリストを上から順に実装します。

1. spec.md を読み込みタスクを抽出する
2. （オプション）タスクをタスク管理ツールに登録する
3. タスクを1つずつ実装する
4. 各タスク後に品質チェックを実行する
5. すべて完了したら `/sdd-review` を促す

---

## 実行手順

### Step 0: 対象の spec.md を決める

`$ARGUMENTS` が指定されていればそのパスを使う。

**指定がない場合:**

```bash
find specs/ -name "spec.md" | xargs ls -t 2>/dev/null
```

結果を新しい順に提示し、番号で選んでもらう:

> `specs/` 内の Spec（新しい順）:
>
> 1. `specs/user-auth/spec.md`（10分前）★ Approved
> 2. `specs/push-notification/spec.md`（昨日）Draft
>
> 番号を選ぶか、別のパスを入力してください。

**ステータスが `Approved` でない場合は警告する:**

> ⚠️ この Spec のステータスは `[現在のステータス]` です。  
> Mob Elaboration（チームレビュー・承認）が完了していない可能性があります。  
> このまま進めますか？（yes / no）

---

### Step 1: タスクリストを抽出して表示する

spec.md の Stage 3 からタスクを抽出し、一覧を表示する:

> **実装するタスク（全 N 件）:**
>
> - [ ] Task 1: [タスク名]
> - [ ] Task 2: [タスク名]
> - [ ] Task 3: [タスク名]（品質ゲート）
>
> タスク管理ツールに登録しますか？

---

### Step 2: タスク管理ツールへの登録（オプション）

次の選択肢を提示する:

> **登録先を選んでください:**
>
> 1. Linear
> 2. GitHub Issues
> 3. Obsidian
> 4. スキップ（登録しない）

選択に応じて以下を実行する。

---

#### 登録先: Linear

1. `list_teams` ツールでチーム一覧を取得して提示する
2. ユーザーがチームを選択する
3. 各タスクを `save_issue` で登録する:
   - **title:** `[spec の機能名] / [タスク名]`
   - **description:** タスクの内容・成功条件・品質チェックコマンドを含める

```
# 登録する内容のイメージ
タイトル: user-auth / LoginScreen UI を実装する
説明:
  ## 内容
  メールアドレス・パスワード入力欄と送信ボタンを実装する

  ## 成功条件
  両フィールドが空の場合、送信ボタンが非活性になる

  ## 品質チェック
  `yarn test LoginScreen`

  ---
  Spec: specs/user-auth/spec.md（/sdd-implement で自動生成）
```

4. 登録完了後、各タスクの Issue URL を表示する

---

#### 登録先: GitHub Issues

`gh issue create` コマンドで各タスクを登録する:

```bash
gh issue create \
  --title "[機能名] / [タスク名]" \
  --body "## 内容\n[タスクの内容]\n\n## 成功条件\n[成功条件]\n\n## 品質チェック\n\`[コマンド]\`\n\n---\nSpec: specs/[機能名]/spec.md"
```

登録完了後、各タスクの Issue URL を表示する。

---

#### 登録先: Obsidian

Obsidian の vault パスを確認する:

> Obsidian の vault パスを教えてください。  
> 例: `~/Documents/MyVault` / `~/Obsidian/Work`

指定されたパスに `[機能名]-tasks.md` を作成し、チェックリスト形式で書き込む:

```markdown
# [機能名] タスクリスト

> Spec: specs/[機能名]/spec.md
> 作成日: [今日の日付]

## タスク

- [ ] Task 1: [タスク名]
  - 内容: [タスクの内容]
  - 成功条件: [成功条件]
  - 品質チェック: `[コマンド]`

- [ ] Task 2: [タスク名]
  ...
```

---

### Step 3: タスクを1つずつ実装する

タスクリストの先頭から順に、以下のサイクルを繰り返す。

**各タスクのサイクル:**

```
① タスクの内容と成功条件を表示する
   ─────────────────────────────
   🔨 Task [N/全N]: [タスク名]
   内容: [タスクの内容]
   成功条件: [成功条件]
   ─────────────────────────────

② 実装する（CLAUDE.md の Steering ルールに従う）
   - Spec の内容に忠実に実装する
   - 仕様から逸脱しそうな場合は実装前に確認する
   - Spec にないことは実装しない

③ 品質チェックコマンドを実行する
   $ [spec.md の品質チェックコマンド]

④ 結果を報告する
   ✅ Task [N] 完了 — [品質チェック結果の要約]
   または
   ❌ Task [N] 失敗 — [エラー内容の要約]
   → 修正してから再度品質チェックを実行する

⑤ 次のタスクに進んでよいか確認する（Human in the Loop）
   > Task [N] が完了しました。Task [N+1] に進みますか？（yes / no / 修正を依頼）
```

**「修正を依頼」と答えた場合:**  
修正内容を聞いて対応してから、品質チェックを再実行して次へ進む。

---

### Step 4: 全タスク完了後の報告

すべてのタスクが完了したら以下を表示する:

> ✅ **すべてのタスクが完了しました。**
>
> **完了サマリー:**
>
> | タスク | 結果 |
> |--------|------|
> | Task 1: [名前] | ✅ 完了 |
> | Task 2: [名前] | ✅ 完了 |
> | Task 3: 品質ゲート | ✅ 通過 |
>
> **次のステップ:**  
> `/sdd-review specs/[機能名]/spec.md` を実行して Mob Construction の準備をしてください。

---

## 注意

- **Spec が Approved でない状態での実装開始は推奨しない**（Mob Elaboration を先に実施する）
- 実装中に Spec の誤りや矛盾を発見した場合は、実装を中断して spec.md の修正を提案する
- 品質チェックが失敗し続ける場合は、3回試みた後に人間に確認を求める
- タスク管理ツールへの登録は実装開始前に行う（実装中に登録すると順序が保証されない）
