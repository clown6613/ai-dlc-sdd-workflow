# /sdd-implement — Spec のタスクを順番に実装する

**使い方:**
- `/sdd-implement specs/<feature>/` — フォルダを指定（`spec.md` を自動で読み込む）
- `/sdd-implement` — 引数なしで実行すると、最近承認された Spec の一覧から選べる

---

## このコマンドがすること

承認済みの `spec.md` を読み込み、タスクリストを上から順に実装します。

1. spec.md を読み込みタスクを抽出する
2. タスクを1つずつ実装する
3. 各タスク後に品質チェックを実行する
4. すべて完了したら `/sdd-review` を促す

---

## 実行手順

### Step 0: 対象フォルダを決める

`$ARGUMENTS` が指定されていれば、そのフォルダを使う（末尾の `/` は任意）。

**指定がない場合:**

```bash
find specs/ -name "spec.md" | xargs ls -t 2>/dev/null
```

結果を新しい順に提示し、番号で選んでもらう:

> `specs/` 内の Spec（新しい順）:
>
> 1. `specs/user-auth/`（10分前）★ Approved
> 2. `specs/push-notification/`（昨日）Draft
>
> 番号を選ぶか、別のパスを入力してください。

**ステータスが `Approved` でない場合は警告する:**

> ⚠️ この Spec のステータスは `[現在のステータス]` です。  
> Mob Elaboration（チームレビュー・承認）が完了していない可能性があります。  
> このまま進めますか？（yes / no）

---

### Step 1: タスクリストを抽出して表示する

確定したフォルダの `spec.md` を Read ツールで読み込む。  
Stage 3 からタスクを抽出し、一覧を表示する:

> **実装するタスク（全 N 件）:**
>
> - [ ] Task 1: [タスク名]
> - [ ] Task 2: [タスク名]
> - [ ] Task 3: [タスク名]（品質ゲート）
>
> 実装を開始します。

---

### Step 2: タスクを1つずつ実装する

タスクリストの先頭から順に、以下のサイクルを繰り返す。

**各タスクのサイクル:**

```
① タスクの内容と成功条件を表示する
   ─────────────────────────────────
   🔨 Task [N/全N]: [タスク名]
   内容: [タスクの内容]
   成功条件: [成功条件]
   ─────────────────────────────────

② 実装する（CLAUDE.md の Steering ルールに従う）
   - Spec の内容に忠実に実装する
   - 仕様から逸脱しそうな場合は実装前に確認する
   - Spec にないことは実装しない

③ 品質チェックコマンドを実行する
   $ [spec.md の品質チェックコマンド]

④ 結果を報告する
   ✅ Task [N] 完了 — [品質チェック結果の要約]
   または
   ❌ Task [N] 失敗 — [エラー内容の要約]
   → 修正してから再度品質チェックを実行する

⑤ 次のタスクに進んでよいか確認する（Human in the Loop）
   > Task [N] が完了しました。Task [N+1] に進みますか？（yes / no / 修正を依頼）
```

**「修正を依頼」と答えた場合:**  
修正内容を聞いて対応してから、品質チェックを再実行して次へ進む。

---

### Step 3: 全タスク完了後の報告

すべてのタスクが完了したら以下を表示する:

> ✅ **すべてのタスクが完了しました。**
>
> | タスク | 結果 |
> |--------|------|
> | Task 1: [名前] | ✅ 完了 |
> | Task 2: [名前] | ✅ 完了 |
> | Task 3: 品質ゲート | ✅ 通過 |
>
> **次のステップ:**
> 1. 仕様との照合レポートを出力する → `/sdd-review specs/[機能名]/`
> 2. チームでレポートをもとに Mob Construction を実施する
> 3. 問題なければ PR を作成してマージ

---

## 注意

- **Spec が Approved でない状態での実装開始は推奨しない**（Mob Elaboration を先に実施する）
- 実装中に Spec の誤りや矛盾を発見した場合は、実装を中断して spec.md の修正を提案する
- 品質チェックが3回連続で失敗した場合は、人間に確認を求める
