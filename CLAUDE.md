# SDD（仕様駆動開発）ワークフロー — Claude Code Steering

このプロジェクトでは **Spec-Driven Development（仕様駆動開発）** を採用しています。
コードを書く前に必ず仕様書（Spec）を作成し、それを AI と人間の共通の「意図の源泉」とします。

---

## ワークフローの全体像

```
1. Intent 作成（人間）
       ↓
2. Spec 生成（AI が提案 → 人間が承認）
   ├── 2-1. 要件（EARS記法）
   ├── 2-2. 設計
   └── 2-3. タスクリスト（SMAV）
       ↓
3. Mob Elaboration（チームレビュー・承認）
       ↓
4. タスク実装（AI が実装 → 人間が確認）
       ↓
5. 品質ゲート（自動チェック）
       ↓
6. Mob Construction（チームレビュー・マージ承認）
```

---

## Spec 生成ルール

### 要件フェーズ（EARS記法）

要件は必ず **EARS（Easy Approach to Requirements Syntax）記法** で記述する。

```
# 基本形（条件あり）
WHEN [条件], the システム SHALL [動作]

# 常に適用
The システム SHALL [動作]

# オプション機能
WHERE [機能が有効な場合], the システム SHALL [動作]

# イベントトリガー
WHEN [イベント] OCCURS, the システム SHALL [動作]
```

**良い例:**
```
WHEN ユーザーがレポートダウンロードボタンをクリックする,
  the システム SHALL CSV ファイルのダウンロードを開始する

WHEN id が存在しない場合,
  the システム SHALL 404 を返す（500 ではない）

The システム SHALL すべての通信を HTTPS で行う
```

**悪い例（曖昧）:**
```
- レポートをダウンロードできること
- エラー処理をする
```

### 設計フェーズ

設計文書には以下を含める:
- **アーキテクチャ概要**: 関与するコンポーネントと責務
- **インターフェース定義**: API エンドポイント、入力/出力の型
- **データフロー**: 処理の流れ（シーケンス図や箇条書き）
- **エラーハンドリング方針**: 各エラーケースの対処法

### タスクリスト（SMAV）

各タスクは **SMAV 原則** に従って記述する:

| 属性 | 意味 | チェック |
|------|------|---------|
| **S**pecific | 「何が」「どう動く」が明確か | "実装する"だけでは不十分 |
| **M**easurable | テストやメトリクスで判定できるか | 数値・条件が書かれているか |
| **A**tomic | 1タスク = 1つのことだけか | 複数の変更を含んでいないか |
| **V**erifiable | 自動または手動で確認できるか | テスト・コマンドが書かれているか |

```
# 良いタスク例
- [ ] GET /api/v1/reports/{id}/csv エンドポイントを実装する
      成功: 200 + CSV ファイル（Content-Disposition: attachment）
      失敗: id 不存在 → 404、60秒超過 → 504
      品質: 単体テスト通過、TypeScript 型エラーゼロ

# 悪いタスク例（曖昧）
- [ ] CSVダウンロード機能を作る
```

## 完了基準の3要素

すべての Spec のタスクには以下の3要素を含める:

```markdown
【成功ケース】
- 正常系の挙動を具体的に記述

【失敗ケース】
- タイムアウト時の挙動
- 不正入力時の挙動
- ネットワーク障害時の挙動

【品質ゲート】
- テスト通過（カバレッジ基準があれば明記）
- 型チェック通過（TypeScript の場合: tsc --noEmit）
- lint クリア（ESLint エラーゼロ）
- セキュリティ要件（必要な場合）
```

## Human in the Loop ルール

以下の操作は **必ず人間の明示的な承認が必要**:

1. **Spec の承認**: 要件・設計・タスクリストを人間がレビューしてから実装開始
2. **本番環境へのデプロイ**: AI が自動でデプロイすることは禁止
3. **データベースの変更**: マイグレーション・データ変更は人間が確認
4. **セキュリティ関連の変更**: 認証・認可・暗号化に関わる実装

## Steering（AI への行動指針）

### コード生成時

- Spec のタスクリストを **上から順に1タスクずつ** 実装する
- 各タスク完了後、品質ゲート（テスト・型チェック・lint）を実行してから次へ進む
- 仕様から逸脱する実装が必要な場合は、**実装前に人間に確認する**
- エラーハンドリングは Spec の「失敗ケース」に従う

### Spec 生成時

- `/sdd-spec` コマンドが呼ばれたら、intent.md を読んで spec.md を生成する
- EARS 記法で要件を記述する
- タスクは SMAV 原則に従い、原子的（1つのことのみ）に分割する
- 不明な点は実装前に質問する（コアメンタルモデルの「質問」ステップ）

### レビュー時

- `/sdd-review` コマンドが呼ばれたら、spec.md と現在の実装を比較する
- 各タスクの完了基準が満たされているか確認する
- 仕様との不整合があれば具体的に指摘する

## テネット（守るべき原則）

| テネット | このプロジェクトでの適用 |
|---------|----------------------|
| **重複の排除** | 仕様書は `specs/` に一元管理。複数箇所に同じ内容を書かない |
| **方法論ファースト** | ツールが変わっても（Cursor に移っても）同じ Spec を使えるよう設計 |
| **再現可能** | Intent + Spec があれば、誰が AI に指示しても同じ品質を実現できる |
| **非依存** | Claude Code 固有の機能に依存しすぎない。CLAUDE.md の内容は他 IDE にも移植可能に |
| **Human in the Loop** | デプロイ・データ変更・セキュリティは必ず人間が承認 |
